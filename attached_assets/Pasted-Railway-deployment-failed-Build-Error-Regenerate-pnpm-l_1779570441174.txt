## Problem
The @workspace/website service (REPO, Railpack builder) fails at the BUILD stage with:
```
ERR_PNPM_LOCKFILE_CONFIG_MISMATCH  Cannot proceed with the frozen installation. The current "overrides" configuration doesn't match the value found in the lockfile
```
The SNAPSHOT stage completed successfully. The build command is `pnpm --filter @workspace/website build`, and the install step runs `pnpm install --frozen-lockfile --prefer-offline` which enforces strict lockfile consistency. The service has watch patterns set to `/artifacts/website/**`.

## Diagnosis
The pnpm-workspace.yaml defines an `overrides` section with many entries (esbuild, rollup, lightningcss, and other platform-specific overrides). A previous Replit Agent commit manually edited pnpm-lock.yaml to add @replit/connectors-sdk instead of running pnpm install, leaving the lockfile's internal overrides hash out of sync with pnpm-workspace.yaml. The --frozen-lockfile flag detects this mismatch and refuses to proceed.

The triggering commit (1ab4329) only changed artifacts/website/public/opengraph.jpg — the lockfile mismatch pre-existed this commit. A fix commit (a1965c60) was pushed to main but was SKIPPED by Railway because the watch patterns (/artifacts/website/**) do not match root-level files like package.json or pnpm-lock.yaml. The package.json on HEAD still contains @replit/connectors-sdk in dependencies, and the lockfile may still be inconsistent.

## Fix
This fix cannot be automated because it requires running pnpm install locally to regenerate the lockfile:

1. Clone the repository and checkout `main`.
2. Run `pnpm install` at the repo root — this regenerates pnpm-lock.yaml with correct overrides metadata matching pnpm-workspace.yaml. If @replit/connectors-sdk should not be a dependency, also remove it from package.json before running install.
3. Commit and push the updated pnpm-lock.yaml (and package.json if changed).
4. After pushing, trigger a manual redeploy from the Railway dashboard (click Deploy on the service), because the watch patterns (/artifacts/website/**) will cause the deployment to be skipped if only root-level files changed.

Optionally, update the service's watch paths to also include /package.json, /pnpm-lock.yaml, and /pnpm-workspace.yaml so future dependency changes automatically trigger deployments.
